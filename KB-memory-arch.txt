вҸә Here's how inbound and outbound files are handled:

  Inbound Flow

  1. Channel extraction вҖ” each channel downloads/extracts attachments
  differently:
  - WhatsApp (src/web/inbound/media.ts) вҖ” downloads via Baileys'
  downloadMediaMessage, resolves MIME types
  - Telegram (src/telegram/bot-message-context.ts) вҖ” uses TelegramMediaRef
  (path, contentType, stickerMetadata) for photos, videos, docs, audio, stickers
  - Discord (src/discord/monitor/message-utils.ts) вҖ” processes file attachments
  and sticker CDN URLs
  - Signal (src/signal/monitor/event-handler.ts) вҖ” extracts inline attachments
  from message bodies

  2. Media storage (src/media/store.ts):
  - Stored under ~/.openclaw/media/inbound/
  - Filename format: {original}---{uuid}.{ext} (collision prevention)
  - TTL: 2 minutes, max 5MB per file
  - downloadToFile() streams HTTP responses to disk with SSRF guard

  3. Message context (src/auto-reply/templating.ts) вҖ” attachments populate
  MsgContext fields: MediaPath, MediaUrl, MediaType (and plural variants for
  multiple attachments), plus Transcript and MediaUnderstanding results.

  4. Sandbox staging (src/auto-reply/reply/stage-sandbox-media.ts) вҖ” copies
  inbound files into <workspace>/media/inbound/ for agent code access, with path
   allowlist validation and symlink escape prevention.

  5. Media understanding (src/media-understanding/) вҖ” runs vision/audio
  analysis, generates MediaUnderstandingOutput[] stored in context.

  6. Media note (src/auto-reply/media-note.ts) вҖ” generates [media attached: ...]
   annotations for the agent prompt (skips already-transcribed audio or
  already-understood media).

  Outbound Flow

  1. Reply payload (src/auto-reply/types.ts):
  ReplyPayload = {
    text?: string;
    mediaUrl?: string;       // single attachment
    mediaUrls?: string[];    // multiple attachments
    audioAsVoice?: boolean;  // voice message vs file
    // ...
  }

  2. Attachment resolution (src/media/outbound-attachment.ts):
  - resolveOutboundAttachmentFromUrl() loads media (local path or URL) вҶ’ saves
  to disk вҶ’ returns { path, contentType }

  3. Delivery pipeline (src/infra/outbound/deliver.ts):
  - deliverOutboundPayloads() orchestrates delivery
  - Text chunked per channel limits (Telegram 4000, Discord 2000 chars)
  - Media resolved via resolveOutboundAttachmentFromUrl()
  - Dispatched through channel-specific outbound adapters

  4. Channel outbound adapters (src/channels/plugins/outbound/):
  - Each channel implements ChannelOutboundAdapter with sendText(), sendMedia(),
   sendPayload()
  - Per-channel implementations in outbound/telegram.ts, outbound/discord.ts,
  outbound/slack.ts, outbound/whatsapp.ts

вҸә Memory Storage & Indexing Architecture

  OpenClaw has a dual-layer memory system: a session persistence layer and a
  semantic memory index.

  ---
  1. Session Persistence

  Storage: ~/.openclaw/agents/<agentId>/sessions/
  - Metadata: sessions.json вҖ” tracks all sessions with token counts, model
  config, origin info, timestamps
  - Transcripts: <sessionId>.jsonl вҖ” line-delimited JSON with {role, content, 
  parentId} messages

  Session keys (src/routing/session-key.ts) uniquely scope conversations:
  agent:{agentId}:{channel}:direct:{peerId}          # per-peer DM
  agent:{agentId}:{channel}:{peerKind}:{peerId}      # group/guild
  agent:{agentId}:subagent:{nestedKey}                # nested agents

  Lifecycle via gateway (src/gateway/server-methods/sessions.ts):
  - sessions.list/preview/resolve вҖ” query & load
  - sessions.patch вҖ” update model/config
  - sessions.compact вҖ” prune old messages, archive to .archived, preserve
  parentId chains
  - sessions.reset/delete вҖ” clear or remove entirely

  ---

вқҜ how to config to use Qmd to replace memory files? 

вҸә Read 1 file (ctrl+o to expand)

вҸә To switch from the builtin memory indexer to QMD, add this to
  ~/.openclaw/config.json5:

  memory: {
    backend: "qmd",
    qmd: {
      includeDefaultMemory: true,   // still indexes MEMORY.md + memory/*.md
      searchMode: "search",         // or "vsearch" (vector), "query"
  (BM25+vector+reranking)
      update: {
        interval: "5m",             // re-index every 5 min
        onBoot: true,               // index on gateway start
      },
      limits: {
        maxResults: 6,
        timeoutMs: 4000,
      },
      // Optional: index extra markdown dirs
      paths: [
        { name: "notes", path: "~/notes", pattern: "**/*.md" }
      ],
      // Optional: also index past conversations
      sessions: { enabled: true, retentionDays: 30 },
    }
  }

  Prerequisites:

  1. Install the qmd CLI: bun install -g https://github.com/tobi/qmd
  2. Ensure qmd is on the gateway's PATH
  3. macOS: brew install sqlite (needs SQLite with extension support)

  Important: QMD doesn't replace the memory markdown files вҖ” they remain the
  source of truth. QMD replaces the search engine that indexes and retrieves
  from those files. The agent still writes to MEMORY.md and memory/*.md; QMD
  just handles the memory_search retrieval instead of the builtin
  SQLite+embeddings indexer.

  If QMD fails or the binary is missing, OpenClaw automatically falls back to
  the builtin MemoryIndexManager.

вқҜ lanceDB can totally replace memory files? 

вҸә Searched for 1 pattern, read 2 files (ctrl+o to expand)

вҸә No, LanceDB cannot fully replace memory files. They serve different purposes:

  What LanceDB does

  - Stores structured memory entries (text + category + importance + vector) in
  its own vector DB at ~/.openclaw/memory/lancedb
  - Auto-capture: after a conversation, scans user messages for trigger patterns
   ("remember", "prefer", emails, phone numbers, etc.) and stores up to 3 per
  conversation
  - Auto-recall: before agent starts, embeds the user's prompt and injects top-3
   matching memories into context as <relevant-memories> block
  - Tools: memory_recall, memory_store, memory_forget

  What it's missing to replace memory files

  1. No file writing вҖ” LanceDB stores opaque vector entries, not readable
  markdown. The agent can't browse, edit, or organize memories like it can with
  MEMORY.md and memory/*.md.
  2. No integration with core memory_search/memory_get вҖ” these tools always go
  through MemoryIndexManager or QmdMemoryManager. LanceDB registers separate
  tools (memory_recall/memory_store) that the core system doesn't know about.
  3. No pre-compaction flush вҖ” the automatic memory flush before context
  compaction writes to markdown files. LanceDB has no hook for this.
  4. Capture is rule-based and limited вҖ” only captures messages matching regex
  triggers (max 500 chars, max 3 per conversation, user messages only). The
  markdown approach lets the agent decide what to store and write arbitrarily
  detailed notes.
  5. No daily logs вҖ” no equivalent of memory/YYYY-MM-DD.md append-only daily
  logs.
  6. No human-readable audit trail вҖ” LanceDB entries are vector DB rows, not
  files you can open and read.

  In practice

  LanceDB is a supplementary layer вҖ” it adds passive auto-recall/auto-capture
  alongside the markdown memory system. The markdown files remain the primary,
  agent-writable, human-readable memory store.

вқҜ then how to choose between qmd and lanceDB 

вҸә They're not competing вҖ” they solve different problems and can run
  simultaneously.

  QMD = better search engine over your existing memory files

  - Replaces the builtin SQLite indexer for memory_search/memory_get
  - Indexes the same MEMORY.md + memory/*.md markdown files
  - Adds BM25 + vector + reranking (vs builtin's hybrid FTS+vector)
  - Runs fully local (no API cost for embeddings)
  - The agent still writes memory to markdown вҖ” QMD just retrieves it better

  LanceDB = passive memory layer that works automatically

  - Separate vector DB alongside everything else
  - Auto-captures facts from user messages without the agent doing anything
  - Auto-recalls relevant memories into context before each turn
  - The agent doesn't need to call any tool вҖ” it happens via lifecycle hooks
  - Useful for "I told you my email last week" type recall

  Decision guide

  Scenario: You want better recall from existing memory files
  Use: QMD

вқҜ without LanceDB, memory_recall/store/forget is implemented in the default 
   settings?                                                                

вҸә Searched for 1 pattern (ctrl+o to expand)

вҸә No. memory_recall, memory_store, and memory_forget only exist in the LanceDB
  plugin. Without it, the only memory tools available are:

  - memory_search вҖ” semantic search over markdown files (builtin or QMD)
  - memory_get вҖ” read a specific markdown file/line range

  Without LanceDB, there's no passive auto-capture, no auto-recall injection,
  and no structured store/forget. The agent remembers things by explicitly
  writing to MEMORY.md or memory/*.md files, and recalls by calling
  memory_search.
